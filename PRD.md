# PRD: Система контроля процессов юридического департамента

**Версия документа:** 1.0
**Статус:** реализовано (MVP)
**Продукт:** веб-приложение «Контроль процессов юридического департамента»

---

## 1. Обзор

Приложение — единый рабочий инструмент юридического департамента для постановки на контроль, отслеживания и закрытия юридических дел: от договоров и претензий до корпоративных процедур и кадровых документов. Заменяет разрозненные таблицы и личные списки задач сотрудников единым реестром с прозрачным статусом каждого дела, историей действий и сигналами по срокам.

Приложением пользуется весь юридический департамент: данные о делах и процессах общие для всех сотрудников, при этом у каждого сохраняется личный профиль (имя) для подписи своих действий.

## 2. Проблема и цель

**Проблема.** Юридический департамент ведёт одновременно несколько разнородных по природе процессов (договоры, иски, заседания правления, кадровые документы, доверенности, ИС и др.). Без единой системы контроля:
- сроки и просрочки не видны руководителю и коллегам;
- история согласований и версии документов теряются или хранятся разрозненно;
- сложно понять, на каком этапе находится конкретное дело и кто за него отвечает.

**Цель.** Дать департаменту единый реестр дел с понятным статусом каждого дела «на сейчас», прозрачным движением по этапам, версиями документов, историей и явной сигнализацией по контрольным срокам — без необходимости вести параллельные таблицы.

## 3. Целевая аудитория

- Юристы департамента — ведут свои дела, продвигают их по этапам, прикладывают документы.
- Руководитель департамента — видит сводную картину по всем процессам, контролирует просрочки и нагрузку.
- Смежные инициаторы (HR, секретариат правления и т.д.) — неявные пользователи: их запросы фиксируются в делах, но напрямую с приложением могут не работать.

## 4. Объём

### Входит в MVP
- Учёт дел по 9 типовым юридическим процессам с возможностью добавлять собственные.
- Полный workflow дела: этапы, документы/версии, история и комментарии.
- Сигнализация по контрольным срокам и просрочкам на уровне всего департамента.
- Дашборд со сводной статистикой.
- Общие данные для всех сотрудников (без авторизации), персональное имя для подписи действий.

### Не входит в MVP (см. раздел 11)
- Авторизация и роли доступа.
- Прикладывание файлов (хранятся только метаданные документов: название, версия, дата, примечание/ссылка).
- Уведомления вне приложения (email, мессенджеры).
- Отчётность/экспорт (PDF, Excel).
- Интеграции с внешними системами (СЭД, почта, календарь).

## 5. Пользовательские роли

Приложение не разделяет пользователей по ролям технически — доступ к данным одинаковый для всех, кто открыл ссылку на приложение. Разграничение «кто за что отвечает» реализовано на уровне данных дела (поле «Ответственный») и личного фильтра «Только мои дела», а не на уровне прав доступа.

## 6. Функциональные требования

### 6.1 Процессы (категории)

Каждый процесс — это шаблон: название, короткое имя для меню, префикс номера дела, цветовой акцент, название стороны/инициатора и упорядоченный список этапов.

Предустановленные процессы:

| Процесс | Префикс | Сумма учитывается | Этапы |
|---|---|---|---|
| Договорно-правовая работа | ДОГ | да | Заявка получена → Проверка контрагента → Подготовка проекта → Юридическая экспертиза → Согласование → Подписание → Регистрация |
| Претензионно-исковая работа | ПИ | да | Поступление → Анализ и оценка риска → Подготовка позиции → Досудебное урегулирование → Подача в суд → Судебное разбирательство → Решение получено → Исполнение / апелляция |
| Сопровождение правления | ПР | нет | Формирование повестки → Сбор материалов → Экспертиза материалов → Проект решений → Заседание проведено → Протокол оформлен → Контроль исполнения |
| Проверка кадровых документов | КД | нет | Поступил на проверку → Правовая экспертиза → Замечания направлены → Доработка → Согласовано |
| Корпоративное управление и регистрационные процедуры | КУ | нет | Поступление запроса → Анализ учредительных документов → Подготовка проекта решения/изменений → Согласование с подразделениями → Утверждение уполномоченным органом → Государственная регистрация изменений |
| Доверенности | ДОВ | нет | Заявка на выдачу → Проверка объёма полномочий → Подготовка проекта → Согласование → Подписание руководителем → Выдача и регистрация в реестре |
| Интеллектуальная собственность | ИС | да | Поступление заявки → Проверка охраноспособности → Подготовка заявки на регистрацию → Подача в уполномоченный орган → Делопроизводство по заявке → Регистрация права → Контроль использования / продление |
| Юридические заключения и консультации | ЗК | нет | Поступление запроса → Анализ вопроса и нормативной базы → Подготовка проекта заключения → Внутреннее согласование → Заключение направлено инициатору |
| Проверка локальных нормативных актов | ЛНА | нет | Поступление проекта документа → Правовая экспертиза → Замечания направлены → Доработка → Согласование → Утверждение и введение в действие |

Требования к управлению процессами:
- Пользователь может добавить новый процесс: название, короткое имя, префикс, цвет, название стороны/инициатора, признак учёта суммы, список этапов произвольной длины.
- Этапы существующего процесса можно переименовывать, добавлять, удалять и переставлять местами; при сокращении списка этапов дела, находившиеся на отсутствующем этапе, автоматически переносятся на последний доступный.
- Процесс нельзя удалить, если по нему есть хотя бы одно дело — система явно сообщает об этом.
- При обновлении приложения новые типовые процессы автоматически добавляются в уже имеющийся у пользователей набор без потери их данных и пользовательских процессов.

### 6.2 Дела (кейсы)

Каждое дело относится к одному процессу и содержит:
- автоматически генерируемый номер вида `ПРЕФИКС-ГОД-0001`, уникальный в рамках процесса;
- название/суть дела;
- наименование стороны/инициатора (подпись поля зависит от процесса);
- ответственного сотрудника;
- приоритет (высокий / средний / низкий);
- дату начала и срок исполнения;
- сумму (если применимо для процесса), в тенге;
- текущий этап;
- признак «на паузе»;
- список документов/версий;
- единую хронологию истории и комментариев.

Действия с делом: создание, редактирование полей, изменение этапа, постановка на паузу/снятие с паузы, удаление (с подтверждением).

### 6.3 Этапы и workflow

Прогресс дела отображается как цепочка этапов («регистрационных штампов»): пройденные этапы отмечены галочкой, текущий выделен, будущие — нет. Любой этап можно открыть кликом — это переводит дело на выбранный этап и автоматически фиксирует переход в истории дела с указанием «откуда» и «куда».

### 6.4 Документы и версии

К делу можно прикреплять записи о документах: название, версия/редакция, дата, примечание или ссылка на место хранения (само приложение не хранит файлы — только метаданные). Добавление и удаление документа отражается в истории дела.

### 6.5 История и комментарии

Единая хронологическая лента по каждому делу объединяет:
- создание дела;
- изменения этапа;
- добавление документов;
- постановку/снятие с паузы;
- произвольные комментарии сотрудников.

Каждая запись подписывается именем сотрудника (из личных настроек) и временем. Добавление комментария доступно через поле ввода в карточке дела.

### 6.6 Контроль сроков и сигнализация

Это ключевое требование продукта: дела с приближающимся или истёкшим сроком должны быть заметны сразу, независимо от того, в каком разделе находится сотрудник.

- **Статусы дела** вычисляются автоматически: «Просрочено» (срок прошёл), «Срок близко» (≤3 дней), «В работе», «Завершено» (последний этап), «Приостановлено» (на паузе).
- **Баннер-сигнал** в верхней части приложения появляется на любом экране, если есть просроченные или горящие дела, и показывает их количество с переходом к полному списку.
- **Раздел «Контрольные сроки»** в боковом меню — отдельный экран со всеми просроченными и горящими делами по всем процессам сразу, разбитый на «Просрочено» и «Срок ≤3 дня», с фильтром «Только мои дела» (по сотруднику).
- **Счётчики в боковом меню**: общий счётчик в разделе «Контрольные сроки» и отдельный счётчик у каждого процесса показывают число просроченных/горящих дел; просроченные визуально приоритетнее (другой цвет).
- **Визуальное выделение строк** в табличном списке дел процесса: просроченные и горящие дела отмечены цветной полосой и цветом срока.
- **Дашборд** показывает агрегированную статистику (всего/просрочено/горит/в работе/завершено/на паузе) и список ближайших по сроку дел с переходом к полному разделу контрольных сроков.

### 6.7 Дашборд

Стартовый экран приложения. Содержит:
- сводные счётчики по всему департаменту;
- список процессов с количеством дел, дел в работе и просроченных по каждому, с переходом в процесс по клику;
- список ближайших по сроку дел (топ-8) со ссылкой на полный список контрольных сроков.

### 6.8 Настройки

- Личное имя сотрудника — используется для подписи действий в истории дел; хранится только у текущего пользователя (не видно другим).
- Явное уведомление о том, что данные по делам и процессам общие для всех, кто открывает приложение.
- Возможность полностью очистить данные приложения (с подтверждением) — для пересборки с нуля.

## 7. Модель данных

**Процесс (категория):** id, название, короткое имя, префикс, цвет, иконка, признак учёта суммы, название стороны/инициатора, упорядоченный список этапов.

**Дело:** id, id процесса, номер, название, сторона/инициатор, ответственный, приоритет, дата начала, срок, сумма, индекс текущего этапа, признак паузы, список документов, список записей истории.

**Документ:** id, название, версия, дата, примечание.

**Запись истории:** id, тип (создание / этап / документ / комментарий / пауза), текст, дата-время, автор.

**Профиль пользователя:** имя (личные данные, не общие).

Данные о процессах и делах хранятся в общем хранилище, доступном всем пользователям приложения; профиль (имя) хранится локально для каждого пользователя.

## 8. UX и визуальный язык

Визуальная метафора — реестр и картотека дел юридического департамента: тёплый «бумажный» фон, акцентные цвета закреплены за смыслом (золотисто-коричневый — договоры, бордовый — иски/риск, синий — корпоративное управление, зелёный — кадровые/рутинные процессы и т.д.), номера дел и даты набраны моноширинным шрифтом как в регистрационных журналах. Прогресс по этапам визуализирован как цепочка проштампованных отметок — единый узнаваемый элемент, повторяющийся везде, где показывается движение дела по процессу.

## 9. Нефункциональные требования

- Приложение работает в браузере без серверной части; данные сохраняются через встроенное хранилище ключ-значение.
- Общие данные (процессы, дела) видны всем пользователям приложения; личные данные (имя) — только владельцу.
- Обновление приложения не должно приводить к потере существующих дел и пользовательских процессов; новые типовые процессы должны добавляться автоматически при следующем открытии.
- Интерфейс должен оставаться читаемым на экранах ноутбука и планшета; боковое меню на узких экранах переходит в горизontальную полосу.

## 10. Ограничения

- Нет авторизации: любой, у кого есть ссылка на приложение, может видеть и изменять все дела и процессы.
- Файлы не хранятся — только метаданные документов (название, версия, дата, примечание/ссылка на внешнее хранилище).
- Нет уведомлений за пределами приложения: сигнал по срокам виден только при открытии приложения.
- Нумерация дел зависит от внутреннего счётчика процесса; при ручном удалении старых дел номера не переиспользуются.

## 11. Дальнейшее развитие (вне MVP)

- Авторизация и разграничение прав (например, право удалять процессы только у руководителя).
- Прикладывание файлов документов, а не только метаданных.
- Уведомления о просрочках по email/мессенджеру.
- Экспорт реестра дел (Excel/PDF) для отчётности.
- Аналитика по нагрузке на сотрудников и срокам прохождения этапов.

## 12. Метрики успеха

- Доля дел с заполненным сроком и ответственным.
- Снижение доли дел, переходящих в статус «Просрочено» без последующего своевременного закрытия.
- Регулярность использования: число активных пользователей в неделю, доля дел с обновлённой историей за последние 7 дней.
- Время от создания дела до первого продвижения по этапу.
